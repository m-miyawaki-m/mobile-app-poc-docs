# プロジェクト指示: mobile-app-poc-bridge

## プロジェクト概要
業務用デバイス連携のための **aar ライブラリ**（デバイス提供元から提供される）を
Capacitor Custom Plugin 経由で JavaScript (Vue) から呼び出せるようにするプラグインを開発する。
Android のみ対応で OK。

## 位置づけ
モバイルアプリ開発環境調査プロジェクトの1リポジトリ。以下5リポジトリのうちの1つ:
- mobile-app-poc-api-spec
- mobile-app-poc-mock
- mobile-app-poc-frontend（このプラグインをインストールして使用）
- **mobile-app-poc-bridge** ← このプロジェクト
- mobile-app-poc-backend

## 技術スタック
- Capacitor 6.x Custom Plugin
- Android: Java or Kotlin
- aar ライブラリ（デバイス提供元提供）
- TypeScript（プラグインのJS側インターフェース）

## 開発環境
- OS: Windows ネイティブ
- Node.js 18+
- Android Studio
- JDK 17+
- エディタ: VS Code + Android Studio
- 社内イントラネット内での開発・配布

```
├─ ios/                           # 使わないが生成される

```

## プラグイン設計指針
- メソッド呼び出し: `@PluginMethod` アノテーション
- 非同期: `call.resolve()` / `call.reject()`
- イベント通知: `notifyListeners("deviceStatus", data)`
- 権限: `@CapacitorPlugin(permissions = ...)`
- フロント側インターフェース例:

```typescript
import { DeviceBridge } from 'mobile-app-poc-bridge';

const result = await DeviceBridge.scan();
DeviceBridge.addListener('deviceStatus', (status) => { ... });
```

## aar 組み込み手順（概要）
1. `android/libs/` 配下に aar 配置
2. `android/build.gradle` に依存追加
```gradle
   dependencies {
     implementation fileTree(dir: 'libs', include: ['*.aar'])
   }
```
3. Plugin クラスから aar の API を呼び出し

## 配布方法（社内に閉じる前提）
- **社内 Git 直接参照**: `"mobile-app-poc-bridge": "git+ssh://internal-git/..."`
- **社内 npm registry**（verdaccio 等を社内に立てる場合）
- **ローカル参照（開発時）**: `"mobile-app-poc-bridge": "file:../mobile-app-poc-bridge"`
- **tarball 配布**: `npm pack` で生成した .tgz を共有フォルダで配布

※ パブリックな npm registry への公開はしない

## 調査観点
- aar の取り込み成功/失敗
- JS ↔ Java のブリッジング作法
- 非同期コールバック・イベント通知の設計
- Logcat でのデバッグフロー
- Capacitor Plugin API の制約
- aar が提供する API の使用感
- 社内配布方法の適否

## 回答方針
- 詳細な回答の前に概要を回答すること
- Capacitor の公式ドキュメント準拠の書き方
- Java コードは Java 17+ 相当の書き方（var, record など積極活用）
- フロント側（TS）とネイティブ側（Java）の両方を意識
- Windows 前提
- デバイス提供元のベンダー名・製品名は極力伏せて「業務用デバイス」「aar 提供元」と表現
- サンプルメソッド名は汎用的に（scan, connect, disconnect, getStatus など）
